## ١. مقدمة عامة ورؤية هندسية للملف (`mikrotik.html`)

يمثل الملف **`MubasherStream/app/.next/server/app/mubasher_admin/mikrotik.html`** واجهة العرض الساكنة مسبقة الرندر (Static Pre-rendered HTML Shell) لـ **صفحة تكامل وإدارة أجهزة وشبكات مايكروتك (MikroTik RouterOS Integration Static View)** لمنصة البث **مباشر ستريم (MubasherStream)** في بيئة الإنتاج تحت المسار الآمن:  
`/mubasher_admin/mikrotik`

يقوم Next.js بتوليد هذا الملف الساكن لضمان عرض واجهات التحكم بالشبكة وخيارات التوجيه والفلترة لمديري المنصة بأسرع شكل ممكن. بمجرد تحميل الإطار الساكن، تنشط معالجات جافا سكريبت لتتم عملية "الترطيب" (React Hydration) لربط كائنات العرض بالخلفية البرمجية، مما يتيح لمسؤولي مباشر ستريم التحكم بجدران الحماية (Firewall rules)، وتوجيه خوادم الـ DNS، وفرض قيود سرعة التحميل لشبكات المشتركين، والتفاعل الفوري مع منافذ راوتر مايكروتك (MikroTik API Ports) لتسيير تدفق قنوات البث بأمان.

## ٢. تفكيك وتحليل محتويات ملف العرض الساكن (`mikrotik.html`)

تتكشف الفحوصات التشريحية البرمجية لمحتويات ملف `mikrotik.html` المترجم عن البنية الهندسية التالية:

1. **روابط تنسيق الواجهة والأصول (Style Sheets & Core Scripts):**
   * ربط الصفحة بملفات التنسيق لـ Tailwind CSS لضمان دقة الألوان وتماسك الواجهة الفني مع السمة الداكنة الكونية للوحة التحكم (Deep Slate-950 Cosmic Theme).
   * إشارات مسبقة التحميل (Preloads) لمحركات ومكتبات الاتصال والفلترة الشبكية لتشغيل الفحوصات الفورية لمعدات التوجيه دون بطء.

2. **بنية الهيكل الرسومي للـ DOM (Document Object Model):**
   * نماذج إدخال معطيات راوتر مايكروتك (Router IP, API Port, Username, Password) مسبقة الرندر بصرياً.
   * الهياكل والجداول المخصصة لعرض القوانين وجدران الحماية (Firewall NAT / Mangle Rules) وإحصاءات سرعة الشبكات الحية.
   * أزرار فحص الاتصال التلقائي (Connection Test Triggers) وتجربة التوجيه.

3. **حقنة بيانات الحالة المبدئية والتهيئة (Dehydrated Initial React State Payload):**
   * يحتوي الملف في آخره على كتل `<script id="__NEXT_DATA__" type="application/json">` المحملة ببيانات JSON والمسؤولة عن تزويد المتصفح ببيانات الأجهزة والشبكات المبدئية وحالة الراوترات المحفوظة، مما يمنع حدوث مشاكل فوارق الترطيب وعرض قراءات عشوائية لحظة تحميل الصفحة.

## ٣. معوقات ملف تكامل راوترات MikroTik الساكن في بيئات Docker وسيرفرات QNAP NAS

عند تشغيل مباشر ستريم ومحاولة التحكم بمعدات مايكروتك عبر واجهة `mikrotik.html` في بيئة حاويات Docker على سيرفرات **QNAP NAS**، تبرز تحديات بالغة الخطورة متعلقة بالعزل الشبكي:

1. **معضلة العزل الشبكي الافتراضي للحاوية (Docker Network Isolation constraints):**
   تعمل حاويات Docker افتراضياً ضمن وضع الشبكة الجسرية المعزولة (Default Bridge Network Mode). في هذا الوضع، تعجز الحاوية برمجياً عن التواصل المباشر مع شبكة كيوناب الفيزيائية (Host Physical Network) التي يتصل بها راوتر MikroTik، مما يمنع نجاح استدعاءات الـ API ويوقف ميزات الفلترة وتوجيه الـ DNS بالكامل.
   * **الحل الفني والترحيل السليم:** يجب تعديل خيارات التجميع داخل ملف Docker Compose واستخدام وضع الشبكة المضيفة (`network_mode: "host"`)، مما يتيح لحاوية مباشر ستريم مشاركة واجهة الشبكة الفيزيائية لسيرفر كيوناب مباشرة والتواصل السلس بغير عوائق مع راوترات مايكروتك.
2. **تضارب منافذ الاتصال والمخاطر الأمنية (Port Conflicts & Security Precautions):**
   قد يؤدي تفعيل وضع الشبكة المضيفة لحظر استخدام بعض المنافذ الافتراضية إذا كانت مستخدمة من قبل تطبيقات أخرى في كيوناب.
   * **توصية الترحيل:** تكوين خيارات حماية جدار الحماية لمايكروتك بدقة، وتعديل منافذ الاتصال الخاصة بـ RouterOS API لتفادي التداخل الشبكي ولتأمين تمرير بايتات القنوات بسلام.
