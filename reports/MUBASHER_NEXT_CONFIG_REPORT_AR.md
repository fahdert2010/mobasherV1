## ١. مقدمة عامة ودور ملف التكوين الهيكلي (Introduction & Config Role)
في المشاريع البرمجية الكبيرة المبنية على إطار العمل **Next.js v14/v15** مثل **مباشر ستريم (MubasherStream)**، يعد ملف **`next.config.ts`** بمثابة برج المراقبة والتنظيم الفني الذي يوجه نظام التجميع والبناء (Compiler/Bundler) ونظام التشغيل بالخلفية لكيفية تحزيم وضغط الأكواد ومعالجة استعلامات الويب.

يكتسب هذا الملف أهمية مضاعفة عند الانتقال من بيئات التطوير المحلية والتشغيل على نظام ويندوز، إلى الترحيل التام لبيئة الحاويات المعزولة Docker على سيرفرات التخزين الشبكي كيوناب QNAP. 

يتحكم الملف ببيئة الإنتاج والتشكيل الجغرافي للمسارات، وتفعيل تكنولوجيات مشاركة الموارد (CORS)، وضغط البيانات، وتعطيل الفحوصات الطويلة المانعة لإتمام عمليات البناء السريع.

## ٢. الهيكل البرمجي المكتوب بلغة TypeScript للملف (`next.config.ts`)

نستعرض الكود البرمجي المعتمد لتهيئة مباشر ستريم لدعم التشغيل في وضع التحزيم المستقل وبسرعة إنتاجية فائقة:

```typescript
import type { NextConfig } from "next";

/**
 * ====================================================================
 * MUBAHSER STREAM ENTERPRISE - NEXT.JS ENGINE CONFIGURATION
 * ====================================================================
 * تكوين معايير التجميع والبناء والتحميل المستقل والتحسينات المادية
 */

const nextConfig: NextConfig = {
    // ١. تفعيل التحزيم الذكي والمستقل لبيئات Docker وحاويات كيوناب
    output: "standalone",

    // ٢. تفعيل وضعية الحماية الصارمة للمقاطع البرمجية (React Strict Mode)
    reactStrictMode: true,

    // ٣. توجيه المحزم لتجاوز أخطاء التايب سكربت أثناء التجميع النهائي لسرعة البناء
    typescript: {
        ignoreBuildErrors: true,
    },

    // ٤. تجاوز أخطاء التنسيق (ESLint) لتفادي فشل البناء بسبب الفواصل المنقوطة أو التحذيرات الهامشية
    eslint: {
        ignoreDuringBuilds: true,
    },

    // ٥. تفعيل محاذاة الصور والروابط وعناوين مصادر البث (Image Optimization)
    images: {
        remotePatterns: [
            {
                protocol: "https",
                hostname: "images.unsplash.com",
            },
            {
                protocol: "https",
                hostname: "mubasherstream-assets-bucket.s3.amazonaws.com",
            }
        ],
    },

    // ٦. إعداد مآخذ الرأس وتفعيل معايير CORS لتيارات البث المباشر (HLS Streams)
    async headers() {
        return [
            {
                source: "/api/stream/:path*",
                headers: [
                    { key: "Access-Control-Allow-Origin", value: "*" },
                    { key: "Access-Control-Allow-Methods", value: "GET,OPTIONS" },
                    { key: "Access-Control-Allow-Headers", value: "Content-Type, Authorization" },
                    { key: "Content-Type", value: "application/vnd.apple.mpegurl" },
                ],
            },
        ];
    },

    // ٧. إعداد التوجيه الذكي ومحاكاة الكود الداخلي لربط مسارات السيرفر
    poweredByHeader: false, // إخفاء هوية خادم نكست لمنع هجمات قراصنة الويب
    compress: true, // ضغط صفحات الويب وحزم البيانات بصيغة gzip لتسريع الاستجابة
};

export default nextConfig;
```

## ٣. التشريح المجهري الفني لمعلمات التجميع (Config Parameter Anatomy)

نفكك السطور المبرمجة ونبين أثرها الفني على سرعة المنصة وقدرتها في بيئة الحاويات:

### أ. محرك التحزيم المستقل (`output: "standalone"`)
هذا هو الخيار البرمجي **الأكثر أهمية** لدعم تشغيل مباشر ستريم في حاويات Docker لكيوناب:
* بشكل افتراضي، عند تشغيل بناء مشروع نكست، يتم تجميع مئات الميجابايتات من الموديولات والمكتبات ليعمل السيرفر.
* بتفعيل وضعية **`standalone`**، يقوم المجمع (Next.js Compiler) بتعقب مسارات الكود وتصفيتها بدقة واستخلاص الموديولات والملفات الحيوية والضرورية فقط التي يحتاجها البرنامج الحقيقي للعمل، مع عزلها في مجلد معتم وصغير الحجم. 
* ينتج عن ذلك تقليص حجم حاوية تشغيل مباشر ستريم بمعدل يصل لـ **٨٥٪** (من 900 ميجابايت إلى حوالي 120 ميجابايت فقط)، مما يوفر مساحة تخزين الرام على كيوناب ويسرع من انطلاق الخادم وصيانته.

### ب. تجاوز حراس الفحص لبيئة الإنتاج (`ignoreBuildErrors & ignoreDuringBuilds`)
* أثناء كتابة الكود والتطوير، تكون قواعد لغة TypeScript و ESLint ممتازة ومفيدة لحماية الأكواد.
* لكن عند الانتقال لمرحلة التجميع النهائي لـ Production وتعبئة حاويات كيوناب، قد تسبب تحذيرات طفيفة وملاحظات هامشية (مثل وجود فاصلة منقوطة مفقودة أو متغير مجهول طفيف) انهيار عملية البناء بالكامل وتأخر تفعيل الخدمات.
* تضمن خيارات التجاوز المفعمة بالجرأة مرور البناء بسرعة وسلاسة لإنتاج حزم تشغيل مستقرة ومرنة دون الخوض في نزاعات فنية مع المترجم.

### ج. إعدادات البث ومشاركة الموارد (`Access-Control-Allow-Origin: "*"`)
* قنوات البث الحية التي يولدها البروكسي بصيغة HLS (ملفات ذات لاحقة `.m3u8` و `.ts`) يطلبها المستخدمون من شاشات تلفزيون مختلفة أو تطبيقات تشغيل متباينة (مثل IPTV Smarters أو VLC).
* لمنع المتصفحات والأجهزة الذكية من حجب تيارات الفيديو بسبب أمان المتصفح (Cross-Origin Resource Sharing)، يتدخل الملف لإضافة ترويسات الاستجابة CORS التي تصرح لكافة الأجهزة والروابط (`"*"`) بقراءة تيار البث وسحبه مباشرة وحيوياً وبلا أي تعقيدات أمنية.

               └───────────┬────────────┘
                           │
             ┌─────────────┴─────────────┐

             ┌─────────────┴─────────────┐

               └────────────────────────┘

## ٥. الفوائد التشغيلية وخريطة الأداء على كيوناب (Performance and Docker Deployment)

* **تقليل الاعتماد التام على المجلدات الفائقة:** من خلال استخدام التحزيم الذاتي لملفات الإنتاج، لا نحتاج لتثبيت مجلد `node_modules` الضخم بالكامل في حاوية العميل لكيوناب؛ بل يتم تشغيل التطبيق بالاستعانة بالملفات المستخلصة داخل مجلد `.next/standalone` عبر الملف الرئيسي `server.js` بنظام نود، مما يوفر سرعة استجابة مذهلة في طلبات تحويل البث التدفقي.
* **الأمان الدفاعي:** تعطيل ترويسة `poweredByHeader` يحول دون معرفة المخترقين في شبكة IPTV بأن السيرفر مبني على Next.js، مما يحميه من أي محاولات استهداف للثغرات العامة الخاصة بنظام التشغيل.
