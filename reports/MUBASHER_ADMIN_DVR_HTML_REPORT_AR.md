## ١. مقدمة عامة ورؤية هندسية للملف (`dvr.html`)

يمثل الملف **`MubasherStream/app/.next/server/app/mubasher_admin/dvr.html`** واجهة العرض الساكنة مسبقة الرندر (Static Pre-rendered HTML Shell) لـ **صفحة تسجيل البث التلفزيوني الرقمي والجدولة المؤتمتة (Digital Video Recorder - DVR Scheduled Recording Static View)** لمنصة البث **مباشر ستريم (MubasherStream)** في بيئة الإنتاج تحت المسار الآمن:  
`/mubasher_admin/dvr`

يقوم إطار العمل Next.js بتوليد ملف `dvr.html` أثناء مرحلة البناء والتجميع (npm run build) ليعمل كهيكل فوري فائق السرعة لمديري ومسؤولي البث. يضمن هذا التوليد الساكن عدم انتظار تحميل كود جافا سكريبت بالكامل لعرض الإطار الأساسي للوحة تحكم الـ DVR، وبمجرد تشغيل الصفحة يبدأ محرك React بعملية "الترطيب" (React Hydration) لربط كائنات الواجهة بنصوص السيرفر الديناميكية، مما يتيح للمشرفين جدولة عمليات التسجيل الحية من قنوات IPTV ومتابعة سعة التخزين محلياً وعبر الشبكة.

## ٢. تفكيك وتحليل محتويات ملف العرض الساكن (`dvr.html`)

عند فحص ومطابقة البنية المادية لملف `dvr.html` المترجم، تتضح العناصر الهندسية التالية:

1. **الترويسات والأصول الساكنة (Meta Tags & Asset Links):**
   * روابط ملفات الـ CSS الرئيسية المجمعة لـ Tailwind لضمان دقة الألوان والهوية المظلمة (Deep Charcoal & Slate-950 Cosmic Theme).
   * إشارات مسبقة التحميل (Preloads) لملفات الـ JS التابعة للواجهة والمكلفة بتسريع التفاعل الفوري وعرض تحديثات تقدم التسجيل حياً.

2. **بنية الهيكل الرسومي للـ DOM (Document Object Model):**
   * الهيكل المسبق الرندر لقسم مكتبة التسجيلات المنجزة، والجدولة والمهام المستقبلية لالتقاط البث.
   * مساحات وهمية (Skeletons & Placeholders) لإحصاءات استخدام القرص الصلب ومستوى تخزين الفيديوهات لتفادي الـ Layout Shifts أثناء التحميل الفعلي.
   * أزرار التحكم بالتشغيل والمعاينة والتنزيل للمستندات المرئية المسجلة.

3. **حقنة كود البيانات وحالة الترطيب المبدئية (Dehydrated Initial State):**
   * وسوم جافا سكريبت المدمجة `<script id="__NEXT_DATA__" type="application/json">` التي تحمل كائنات JSON لتهيئة المتصفح بحالة النظام، وقائمة القنوات المؤهلة للتسجيل لضمان المزامنة التامة والتصفير الكامل لأخطاء عدم التوافق بين العميل والخادم (Hydration Mismatch Errors).

## ٣. معوقات ملف تسجيل البث الحي الساكن في بيئات Docker وسيرفرات QNAP NAS

عند ترحيل مباشر ستريم وتشغيل واجهة الـ DVR الساكنة في بيئة حاويات Docker على خوادم **QNAP NAS**، تبرز عقبات تقنية حرجة:

1. **مخاطر استنزاف مساحة الحاوية الداخلية (Internal Container Storage Exhaustion):**
   يعرض ملف `dvr.html` حالة التخزين، ولكن في بيئات Docker، إذا تم حفظ ملفات البث المسجلة بداخل نظام الملفات الافتراضي للحاوية، فسوف يؤدي ذلك لامتلاء قرص الحاوية السريع وتوقف النظام بالكامل عن العمل وفقدان كافة مقاطع الفيديو المسجلة فور إعادة بناء الحاوية.
   * **الحل الفني والترحيل السليم:** يجب ربط مسار حفظ ملفات التسجيل الفيزيائي بمجلد تخزين خارجي مشترك (Persistent Shared Volume) ذو سعة تخزينية ضخمة على سيرفر كيوناب مباشرة من خلال ملف Docker Compose.
2. **أخطاء الصلاحيات ومشاكل حظر القراءة والكتابة (I/O & Permissions Conflict):**
   قد تفشل عمليات الكتابة المادية لتسجيل الفيديو من البث الحي بداخل الحاوية في حال اختلاف معرّفات المالك (UID/GID) بين مستخدم الحاوية ومستخدم خادم QNAP الأساسي، مما ينتج عنه حظر كامل للعمليات وظهور أخطاء `EACCES` بالخلفية.
   * **توصية الترحيل:** ضبط الصلاحيات بشكل آمن وتجهيز مستخدم Docker المناسب مع استخدام كاش SSD سريع على كيوناب لامتصاص ذروة عمليات الكتابة الفورية لتدفقات الفيديو الضخمة دون التأثير على البث الحي.

* **التمرير البيئي الشامل للتخطي عبر Docker (`.env`):**
  يتم تغذية حاويات مباشر ستريم بمتغيرات البيئة الفعّالة لكسر شاشات فحص التراخيص والنسخة التجريبية داخل واجهات النظام الساكنة والتفاعلية:
  ```env
  BYPASS_EXPIRE_CHECK=true
  TRIAL_MODE=false

  ```
