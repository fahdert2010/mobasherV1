## مقدمة المرحلة الأولى
تستهدف هذه المرحلة تنظيف بيئة التطوير الحالية من كافة الملفات الثنائية (Binary Executables) ومحركات التسيير (Batch Scripts) التي صممت خصيصاً للعمل على نظام التشغيل ويندوز، وتهيئة مجلدات كود خادم مباشر ستريم برمجياً ونظامياً ليكون جاهزاً للعمل على بيئة Linux المعزولة داخل حاوية Docker في كيوناب دون حدوث أخطاء تعارض برمجية.

## الجزء الأول: جانب الكود وحصر الملفات والتعديلات برمجياً (Code & File Purge)

### ١. حصر ملفات متبقيات ويندوز المراد استبعادها وحذفها فكرياً من دورة التشغيل:
* **الملف الأول:** `/MubasherStream_Target_System/service/nssm.exe`
  * **الوظيفة المسببة للمشكلة:** ملف تنفيذي خارجي للويندوز يُستخدم لتثبيت السيرفر كخدمة ويندوز خلفية (Windows Service Manager).
  * **التأثير في لينكس:** ثنائي غير متوافق تماماً (Unsupported x86 Windows Binary) ويؤدي لمحاولات تشغيل فاشلة وانهيار السيرفر عند الإقلاع في Docker.
* **الملف الثاني:** `/MubasherStream_Target_System/scripts/install-service.js`
  * **الوظيفة المسببة للمشكلة:** سكربت لغة جافا سكربت يستدعي أداة `nssm.exe` لتسجيل الخدمة في الويندوز.
  * **التأثير في لينكس:** محاولات استدعاء مفقودة لملفات مفقودة وتشتيت لكود الخادم.
* **الملف الثالث:** `/MubasherStream_Target_System/runtime/nodevars.bat`
  * **الوظيفة المسببة للمشكلة:** ملف تهيئة الأوامر الدفعية لويندوز لتهيئة متغيرات بيئة نود المحمولة.
  * **التأثير في لينكس:** كود دفعي غير مقروء بالكامل في لينكس.
* **الملف الرابع:** `/MubasherStream_Target_System/runtime/install_tools.bat`
  * **الوظيفة المسببة للمشكلة:** ملف تحميل وتثبيت أدوات المطورين لويندوز.
  * **التأثير في لينكس:** غير ضروري ويعيق خفة مساحة مجلد السيرفر.

### ٢. التعديلات المطلوبة في الكود بالتفصيل:

#### أ. تعديل ملف الحزم والتوجيه المصدري (`package.json`)
* **اسم الملف ومساره:** `/MubasherStream_Target_System/package.json`
* **السطر المستهدف:** أسطر السكربتات المخصصة لويندوز (السطور **١٢ إلى ١٥**).
* **الكود قبل التعديل (النسخة الأصلية للويندوز):**
  ```json
    "scripts": {
      "start": "node server.js",
      "install-service": "node scripts/install-service.js",
      "uninstall-service": "node scripts/uninstall-service.js"
    }
  ```
* **الكود بعد التعديل (النسخة النظيفة المجهزة للينكس ودوكر كيوناب):**
  ```json
    "scripts": {
      "start": "node bootstrap.js"
    }
  ```
* **أثر التعديل:** إزالة أي محاولات لاستدعاء سكربتات تثبيت الخدمات الخلفية لويندوز وتوجيه محرك الإقلاع مباشرة لمغلف التفعيل الذكي `bootstrap.js` المعزز.

## الجزء الثاني: جانب الكيوناب وإرشادات النقل والتهيئة (QNAP Prep & Guidelines)

### ١. تجهيزات وإعدادات نظام كيوناب (NAS Workspace Setup):
* **تجهيز المجلد المؤقت في QNAP:** قبل الشروع في بناء أو نقل أي حاويات، يتم الدخول إلى نظام التشغيل **QTS** الخاص بكيوناب، والتوجه إلى **File Station**، والتأكد من تفعيل بروتوكول نقل الملفات **SFTP** أو شبكة **Samba / Windows Networking** في لوحة التحكم (Control Panel -> Network & File Services -> Win/Mac/NFS/WebDAV).
* **صلاحيات الوصول:** تعيين حساب مستخدم فريد (مثال: `mubasher_admin`) ومنحه صلاحيات القراءة والكتابة والوصول الكامل لمجلد حاويات كيوناب الافتراضي `/share/Container`.

### ٢. إرشادات وقنوات نقل الملفات النظيفة لكيوناب:
* **تجميع الملفات النظيفة (Packaging):** 
  * في بيئة التطوير الحالية، يتم تجميع مجلد التطبيق بعد استبعاد الملفات الأربعة المذكورة في الجزء الأول.
  * **هام جداً:** يجب استبعاد مجلد `node_modules` بالكامل من عملية النقل؛ حيث إن الحزم المبنية على ويندوز (Windows native bindings) لن تعمل على نظام لينكس وسوف تسبب أخطاء ربط ثنائية (Native module mismatch errors). سيقوم محرك Docker بتحميل الحزم النظيفة والمتوافقة مع لينكس تلقائياً من ملف `package.json` عند بناء الصورة.
* **آلية النقل:**
  * ضغط مجلد `MubasherStream_Target_System` النظيف في ملف مضغوط باسم `mubasher_clean.zip`.
  * رفع الملف باستخدام بروتوكول **SFTP** عبر برنامج **FileZilla** أو مباشرة عبر سحب وإفلات الملف في واجهة **File Station** الخاصة بكيوناب إلى المسار المؤقت: `/share/Container/mubasher_temp`.
