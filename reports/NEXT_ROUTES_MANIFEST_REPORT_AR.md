## ١. مقدمة عامة ومعمارية الروابط وعلاقتها بـ RegEx (Introduction & Routing Engine)
عندما يطلب متصفح المستخدم رابطاً معيناً من خادم مباشر ستريم (مثل لوحة تحكم الإدارة `/mubasher_admin` أو تحديثات قنوات البث `/api`)، فإن الخادم والواجهة الأمامية بحاجة إلى نظام معالجة فوري يقوم بـ:
1. **مطابقة الروابط بدقة متناهية (URL Matching):** التعرف على طبيعة الرابط المكتوب وهل يتبع الروابط الساكنة أم يحتاج لمعاملات فرعية ديناميكية.
2. **التعامل مع الترويسات وإعادة التوجيه (Headers, Redirects & Rewrites):** تطبيق قواعد الأمان وتمرير الترويسات الخاصة ببروتوكول React Server Components (RSC) بصفر مجهود معالجة.

لتحقيق هذه المهمة بكفاءة مطلقة، يعتمد محرك Next.js على ملف المانيفست المركزي للروابط المسمى **`routes-manifest.json`** ويقع في المسار:
`.next/routes-manifest.json`

يضم هذا الملف الخارطة الكلية للروابط مدعومة بحسابات التعابير النمطية (Regular Expressions - RegEx) لتوفير مطابقة سريعة للغاية وفورية على مستوى خادم نود والواجهة الأمامية للموقع.

## ٢. الهيكل البرمجي الكامل والفعلي لملف المانيفست (`routes-manifest.json`)

في تطبيق مباشر ستريم المطور للإنتاج، تتم صياغة ملف خارطة الروابط كالتالي:

```json
{
  "version": 3,
  "pages404": true,
  "caseSensitive": false,
  "basePath": "",
  "redirects": [],
  "headers": [],
  "rewrites": [],
  "staticRoutes": [
    {
      "page": "/",
      "regex": "^/(?:/)?$",
      "routeKeys": {},
      "namedRegex": "^/(?:/)?$"
    },
    {
      "page": "/mubasher_admin",
      "regex": "^/mubasher_admin(?:/)?$",
      "routeKeys": {},
      "namedRegex": "^/mubasher_admin(?:/)?$"
    },
    {
      "page": "/api",
      "regex": "^/api(?:/)?$",
      "routeKeys": {},
      "namedRegex": "^/api(?:/)?$"
    }
  ],
  "dynamicRoutes": [],
  "rsc": {
    "header": "RSC",
    "varyHeader": "RSC, Next-Router-State-Tree, Next-Router-Prefetch",
    "prefetchHeader": "Next-Router-Prefetch",
    "contentTypeHeader": "text/x-component"
  }
}
```

## ٣. التحليل والتشريح المجهري لكل سطر ومفتاح (Key-by-Key Anatomical Breakdown)

نفكك الآن تفاصيل الكود، المسافات، والمعاني الهندسية المخصصة لكل قسم في الملف بدقة مجهرية:

### أولاً: كائن التهيئة والخصائص العامة (السطر ١ - ٧)
* **السطر ١:** `{`
  * **الوظيفة الفنية:** بدء تعريف كائن JSON المركزي لتجميع الروابط.
* **السطر ٢:** `"version": 3,`
  * **الوظيفة الفنية:** يثبت معيار وإصدار ملف الـ Routes Manifest الذي يتوقعه محرك تشغيل خادم نكست.
* **السطر ٣:** `"pages404": true,`
  * **الوظيفة الفنية:** يعلم الخادم بوجود صفحة خطأ 404 مخصصة مبنية مسبقاً لعرضها للمستخدم فورياً في حال طلب روابط عشوائية غير مسجلة لدينا.
* **السطر ٤:** `"caseSensitive": false,`
  * **الوظيفة الفنية:** يحدد أن الروابط لا تتأثر بحالة الأحرف (غير حساسة). على سبيل المثال، طلب الرابط `/Mubasher_Admin` سيتم توجيهه ومعالجته بنفس طريقة الرابط المكتوب بحروف صغيرة `/mubasher_admin`.
* **السطر ٥:** `"basePath": "",`
  * **الوظيفة الفنية:** يحدد مسار الرابط الأساسي للموقع (Prefix). بما أنه فارغ، فإن الموقع يعمل مباشرة في جذر النطاق الفعلي دون مجلدات فرعية خارجية بالرابط.

### ثانياً: إعادة التوجيه والترويسات والتحويل الخلفي (السطر ٦ - ٨)
* **السطر ٦ - ٨:**
  ```json
  "redirects": [],
  "headers": [],
  "rewrites": []
  ```
  * **الوظيفة الفنية:** مصفوفات لتخزين أي قواعد مخصصة لإعادة التوجيه (Redirects) أو الترويسات الأمنية (Security Headers) أو التحويل الخلفي الصامت للروابط (Rewrites). تظهر فارغة نظراً لعدم إدخال تعديلات مخصصة عليها في ملف الإعداد الرئيسي وتفضيل إدارة الأمان عبر الواجهات بشكل مباشر.

### ثالثاً: تشريح الروابط الثابتة والتعابير النمطية الحارسة `"staticRoutes"` (السطر ٩ - ٢٧)
* **السطر ٩:** `"staticRoutes": [`
  * **الوظيفة الفنية:** الفهرس الأكثر أهمية لتسجيل الروابط الثابتة الصالحة في موقع مباشر ستريم ومطابقتها الفورية باستخدام RegEx لضمان سرعة معالجة مذهلة.
* **السطر ١٠ - ١٥: فحص مطابقة الصفحة الرئيسية (`"/"`)**
  ```json
  {
    "page": "/",
    "regex": "^/(?:/)?$",
    "routeKeys": {},
    "namedRegex": "^/(?:/)?$"
  }
  ```
  * **الوظيفة الفنية والـ RegEx:**
    * **`"page": "/"`**: الرابط المستهدف.
    * **`"regex": "^/(?:/)?$"`**: التعبير النمطي الحارس لمطابقة الصفحة الرئيسية. دعونا نشرحه مجهرياً:
      - `^`: يعني بدء نص الرابط.
      - `/`: يطابق المائل المائل للأمام الأساسي.
      - `(?:/)?`: مصفى لالتقاط أي شرطة مائلة نهائية (Trailing Slash) اختيارياً دون تذكرها كمعامل فرز، مما يجعل الرابطين `http://domain.com` و `http://domain.com/` متطابقين تماماً وموجهين لنفس المكون بسلام.
      - `$`: يعني نهاية نص الرابط تماماً لمنع مطابقة روابط أطول عشوائية.
* **السطر ١٦ - ٢١: فحص مطابقة صفحة الإدارة (`"/mubasher_admin"`)**
  ```json
  {
    "page": "/mubasher_admin",
    "regex": "^/mubasher_admin(?:/)?$",
    "routeKeys": {},
    "namedRegex": "^/mubasher_admin(?:/)?$"
  }
  ```
  * **الوظيفة الفنية:** يطابق التعبير النمطي `^/mubasher_admin(?:/)?$` رابط إدارة مباشر ستريم بدقة مطلقة، ويسمح بوجود شرطة مائلة إضافية في نهايته اختيارياً.
* **السطر ٢٢ - ٢٧: فحص مطابقة واجهة الـ API الخلفية (`"/api"`)**
  * **الوظيفة الفنية:** يطبق نفس آلية المطابقة الصارمة والسريعة للطلبات الموجهة لواجهات البيانات.

### رابعاً: معايير بروتوكول تدفق وحفظ رياكت `"rsc"` (السطر ٢٩ - ٣٤)
* **السطر ٢٩:** `"rsc": {`
  * **الوظيفة الفنية:** يحدد ترويسات ومواصفات بروتوكول React Server Components (المعروف بـ RSC) والذي يستخدمه نكست جي إس لتبادل البيانات غير المتزامنة وتدفق الواجهات بين العميل والخادم:
    * **`"header": "RSC"`**: الترويسة المخصصة المرسلة في طلبات المتصفح ليعلم الخادم برغبته في جلب بيانات RSC فقط بدلاً من رسم كود HTML كامل.
    * **`"varyHeader": "RSC, Next-Router-State-Tree, Next-Router-Prefetch"`**: ترويسة إرشادية للمتصفحات والـ Proxy (مثل CDN وكاش كيوناب) لإعلامها بضرورة عزل كاش الـ RSC عن كاش طلبات الويب العادية لمنع تداخل البيانات وعرض كود خام للمشاهدين.
    * **`"contentTypeHeader": "text/x-component"`**: نوع المخرجات المرتجعة للبيانات والتي يعالجها محرك رياكت في المتصفح لرسم المكونات ديناميكياً.

## ٤. مخطط مطابقة وتوجيه حركة مرور الويب (URL Matching RegEx Lifecycle)

يوضح المخطط التالي دورة حياة مطابقة الرابط المطلوب باستخدام قواعد التعابير النمطية في المانيفست:

```text
       ┌────────────────────────┐
       │   وصول طلب رابط جديد  │  <── (مثال: طلب رابط /api)
       └───────────┬────────────┘
                   │
                   ▼ (البدء في مطابقة الـ RegEx)
       ┌────────────────────────┐
       │  routes-manifest.json  │  <── (مسح مصفوفة staticRoutes خطوة بخطوة)
       └───────────┬────────────┘
                   │
         ┌─────────┴────────────────────────┐
         ▼ (هل يطابق التعبير النمطي للرابط؟)  ▼ (لم يتم العثور على أي مطابقة؟)
┌─────────────────────────────────┐   ┌─────────────────────────────────────┐
│  السماح بالمرور وتوجيه الطلب     │   │  ١. التحقق من وجود "pages404"      │
│  للمكون الفعلي "/api/route.js"  │   │  ٢. عرض واجهة خطأ 404 للمستخدم      │
└─────────────────────────────────┘   └─────────────────────────────────────┘
```

## ٥. المميزات الفنية والمواصفات الجمالية للملف (Technical Strengths)

* **السرعة المطلقة للمطابقة:** بفضل التعابير النمطية المسبقة الحساب (Pre-compiled RegEx)، تصبح عملية البحث وتوجيه الروابط تتم بـ Microseconds (أجزاء من المليون من الثانية)، مما يجعل أداء مباشر ستريم فائق السرعة والخفة.
* **المرونة والتوافقية الكاش لـ RSC:** تضمن إعدادات الـ RSC التنسيق التام بين محركات التخزين الشبكي ومتصفحات المستخدمين دون أخطاء أو تداخل في الكاش الرقمي.
* **نظام التباعد وهيكلية الملف:** يعتمد التنسيق الزوجي الدقيق (2-space Tab size) بنهاية أسطر LF المتوافقة تماماً مع معايير البناء المستمر لبيئات السيرفرات والشبكات.
